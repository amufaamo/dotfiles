## Step 1: タスク分析（Leadが最初に必ず実行）

タスクを受け取ったら、チームを組む前に以下を自問すること：

### 1-A. 作業種別は何か？
- **実装系**: コード・インフラ・設定を作る
- **調査系**: 情報収集・文献調査・技術検証
- **執筆系**: 論文・レポート・ドキュメント・記事
- **分析系**: データ解析・ログ調査・パフォーマンス計測
- **レビュー系**: コード・文章・設計の評価

### 1-B. 並列化できるか？
- 独立した「関心事」が3つ以上ある → Team有効
- 前の作業が終わらないと次に進めない → Team不要
- 同じファイル・文書を複数人が同時編集 → Team不要

### 1-C. 規模の判定
- 関心事が1つ（単一ファイル・単一テーマ） → 単独
- 関心事が2〜3つ、互いに独立している → 2〜3名
- 関心事が4つ以上、複数ドメインにまたがる → 4〜5名（過剰スポーン禁止）

## Step 2: 協調スタイルの選択

| スタイル | 使う場面 |
|---------|---------|
| **並列独立型** | 各自が独立したパートを担当→最後にLeadが統合 |
| **競合仮説型** | 同じ問いに複数エージェントが異なる仮説で挑む |
| **順番リレー型** | 調査→執筆→校正のような依存関係がある作業 |

---

## Step 3: ロール定義集

### 【実装系】
- **implementer**: コード実装全般。変更したAPIは即座にfrontendへ通知
- **tester**: テスト作成・実行。実装完了宣言後に動き始める
- **infra**: インフラ・CI/CD。環境変数追加時は全員にブロードキャスト

### 【調査系】
- **researcher**: 特定テーマの情報収集。情報源・信頼性を明記して全員に共有
- **investigator**: 競合仮説の検証。独立した仮説を持ち積極的に反証する

### 【執筆系】
- **writer**: 担当セクションの執筆。担当範囲を宣言してから開始
- **editor**: 全体の論旨・一貫性・文体を統一。writer完了後に動く
- **fact-checker**: 記述の正確性・引用の確認

### 【分析系】
- **analyst**: データ・ログ・メトリクスの解析
- **visualizer**: 分析結果のグラフ・図表化

### 【品質系】（どのプロジェクトにも適用可）
- **reviewer**: 成果物全体の品質チェック（コード/文章/分析それぞれに対応）

## Step 4: プロジェクト種別ごとの推奨編成

| プロジェクト種別 | 推奨チーム | 協調スタイル |
|----------------|-----------|-------------|
| 新機能開発 | implementer + tester + reviewer | 並列独立 |
| バグ・原因調査 | investigator × 3 | 競合仮説 |
| 論文・レポート執筆 | researcher × N + writer × N + editor | 並列独立→リレー |
| 技術選定・比較調査 | researcher × N + reviewer | 競合仮説 |
| データ分析レポート | analyst + visualizer + writer | 順番リレー |
| コードレビュー | reviewer × 3（観点別）| 並列独立 |
| 単純な修正・小タスク | Team不要 | — |

## Step 5: 共通ルール（全エージェント必読）

### 作業範囲の宣言
スポーン直後に担当範囲を宣言してから開始する

### 他エージェントへの通知が必要な場面
- 共有リソース（型定義・設定・参考文献リスト）を変更するとき
- 自分の担当が他のエージェントに影響するとき
- 想定外の発見があったとき

### 完了報告の書き方
「何を・どこで・どう確認したか」を必ず含める
- 実装: 「〇〇を実装。テストXX件通過を確認」
- 執筆: 「第3章XX字を執筆。引用YY件を文献リストに追加」
- 調査: 「テーマXXを調査。主要文献ZZ件を要約、信頼性AA」

### Leadの動き方
- チームを組んだらDelegate modeに切り替える（Shift+Tab）
- スポーンプロンプトに必ずコンテキストを含める（全員がゼロから始まる）

## 【学習済みルール】

### 2026-03-14 HostKeeper開発から
- Flutter Web の起動は `nohup flutter run -d web-server --web-port 8081` を使う（`&`だけでは即終了する）
- GoRouter では子画面・詳細画面への遷移に `context.push()`、ルート切替（ログイン→ダッシュボード等）に `context.go()` を使い分ける
- デモモード導入時は `grep -r "authStateProvider\|FirebaseAuth"` で Firebase依存を全洗い出ししてから対応する

### 2026-03-18 SingleRepExplorer / Docker + Shiny開発から
 - Docker内Shinyのモジュールコードを変更したら `pkill -HUP` ではなく **`docker restart <container>`** で完全再起動（Rワーカープロセスが新コードをロードするのはプロセス再起動時のみ）
 - Windows環境では **`docker exec -it` は失敗**する（non-TTY）。必ず `docker exec -i` のみ使うこと
 - Shinyアプリの `renderPlot()` 出力キャンバスは Chrome MCP の **`zoom` ツールで白く返る**。plot取得には `screenshot` を使うこと
 - コンテナへの大容量ファイル転送は Chrome MCP `file_upload` ではなく **`docker cp`** を使うこと
 - Shinyクラッシュの根本原因は **`/var/log/shiny-server/*.log`** を `ls -lt` で最新ファイルを特定し `grep "Warning: Error"` で調べる
 - R Shinyで「could not find function」エラーが出たら **namespace衝突**を疑う（例: `immunarch` が `data.table` をロードして `:=` を上書き、`S4Vectors` が `dplyr::first()` を上書き等）

### 2026-03-19 WSL / Google Drive (G:) マウントトラブル
 - WSLから `/mnt/g` にアクセスした際に `failed 19 (ENODEV)` が出た場合はマウントが解除されている。**`sudo mount -t drvfs G: /mnt/g`** で手動マウントすること
 - `git clone` をGoogle Drive内のディレクトリに行う際、WSL側からだとパーミッションエラーで失敗することが多いため、**Windows側のPowerShell**等で実行するのが確実

## Step 7: project.md ルール（全プロジェクト共通）

プロジェクトのルートに `project.md` が存在する場合、それを**単一の真実の情報源（Single Source of Truth）**として扱うこと。

### 作業前の必須確認
新しいタスクや会話を始める際は、必ず最初に `project.md` を読み込み、
プロジェクトの目的・前提ルールを把握した上で作業を開始する。

### 自律的な自動更新
以下のタイミングで、明示的な指示がなくても `project.md` を更新する：
- タスクが完了したとき → 「4. 現在のタスク」を更新
- 重要な決定事項があったとき → 「5. 活動履歴」に時系列で追記
- 過去の重要な文脈は消さない

### 構造の維持
`project.md` は必ず以下の構成を維持し、勝手に項目を削除・フォーマット変更しない：
1. 目的と背景 (Why)
2. ゴールと最終成果物 (What)
3. ルール・制約・前提条件 (How)
4. 現在のタスクとスケジュール (Action)
5. 活動履歴・決定事項 (Log)
